SECTION C1 — Atomic migration execution  (audit: F1-05)
═══════════════════════════════════════════════════════════════
File: src/db/migrations/index.ts  →  runMigrations function.

Current behavior: each migration is executed by
`await dbClient.run(migration.sql)`. If a multi-statement SQL fails midway,
the partially-applied statements persist, version is NOT updated, and the
next startup re-runs the migration — which usually fails on "table already
exists" and leaves the DB unusable.

Required change:
1. Wrap each migration execution in a transaction that includes the
   setVersion call. Pseudocode:

     for (const migration of migrations) {
       if (migration.version > currentVersion) {
         try {
           await dbClient.run('BEGIN TRANSACTION');
           await dbClient.run(migration.sql);
           // setVersion is via PRAGMA user_version — include INSIDE tx
           await dbClient.run(`PRAGMA user_version = ${migration.version}`);
           await dbClient.run('COMMIT');
         } catch (err) {
           try { await dbClient.run('ROLLBACK'); } catch (_) {}
           console.error(`Migration ${migration.version} failed:`, err);
           throw err;
         }
       }
     }

2. IMPORTANT: some existing migration .sql files already contain `PRAGMA
   user_version = N;` at the top. That's fine — it'll be executed inside
   the transaction. The explicit `PRAGMA user_version = ${version}` after
   the SQL ensures the version is set even if the file forgets to.
3. Verify SQLite-WASM supports `BEGIN TRANSACTION` over multi-statement
   exec. If `dbClient.run` cannot handle a standalone BEGIN/COMMIT, use
   the multi-statement form in a single call:
     await dbClient.run(`BEGIN TRANSACTION; ${migration.sql};
                         PRAGMA user_version = ${v}; COMMIT;`);
4. Keep `dbClient.setVersion(...)` for compatibility if other code uses it.
